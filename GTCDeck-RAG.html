01 — The Bottleneck
~2 min

The GPU Is Not Waiting for Compute. It's Waiting for Data.

In RAG, retrieval happens during inference — not offline. Every query performs multiple data operations before the model generates a single token.

Typical RAG inference flow — every user query:
Query
→
Embed
→
Vector Search
→
Metadata
→
Object Retrieval ⚠
→
Assemble
→
LLM ✓
Storage latency sits on the critical path to inference. Each query can require multiple reads: metadata objects, document chunks, embeddings, context windows.

Even tens of milliseconds per operation accumulate quickly under concurrent load.
The consequence: To maintain the same response time target, you must either add more GPUs, accept longer response times, or reduce workload concurrency.

That's an infrastructure problem, not a model problem.
TTFT is driven by
① Query embedding — GPU, fast
② Vector lookup — index-bound
③ Storage read — bottleneck
④ Prompt assembly — CPU, fast
⑤ First token generated
Common assumption vs reality
Most enterprise RAG deployments assume storage and compute are loosely coupled — a reasonable design for durability, but it introduces an additional network hop on every inference request.
Under load, that hop becomes visible.

03 — Behaviour Under Load
~2 min 144 chunks · 50 concurrent readers · 7,200 total reads · Live AWS data

The Most Important Result is at Concurrent Load.

At ~50 concurrent read requests, DDN Infinia remained stable at ~6ms. S3 climbed to 706ms. This is where retrieval latency becomes a throughput problem, not just a response time problem.

04 — What This Means
~2 min

Improving the Data Layer Improves the Entire System's Operating Envelope

When retrieval latency becomes predictable and low, several improvements become possible — without changing a single line of model code.

System Performance
→More consistent response times under load
→Higher throughput per GPU within the same SLO
→Latency stability as concurrency increases
→Better TTFT for interactive AI applications
Infrastructure Economics
→Lower queries per GPU forces overprovisioning
→More hardware needed just to meet peak demand
→Modelled at 1K GPUs: ~$1.72M/mo in idle compute if 60% is retrieval-bound
Modelled projection — not measured directly
Real savings depend on workload, utilization, and batching strategy
Scalability
→DDN Infinia held ~6ms at 50 concurrent readers
→S3 reached 706ms at same load — and degrades further
→Latency stability under concurrency is the key variable at scale
Production performance depends on query distribution, cache, prompt length, batching
